1 · Introduction & Overview

1.1 What is AURA?

AURA (Autonomous Understanding and Response Architecture) is a real-time predictive maintenance platform purpose-built for the International Space Station's Environmental Control and Life Support System (ECLSS). It combines a physics-based sensor data generator, a four-layer machine learning pipeline, interactive 3D visualization, and a natural-language AI analyst into a single browser-accessible application.

AURA monitors seven ISS modules continuously, applies anomaly detection and fault classification to every reading, predicts failure probability and remaining useful life, and recommends remediation actions — all in real time, updating every second.

1.2 System Capabilities

◈

⟳

Physics-correlated data generator simulates 20 ECLSS parameters per location with configurable noise and fault drift.

⚙
Secured Settings

2.2 Starting on Linux

  1. Open a terminal and navigate to the project root directory.
  2. Make the start script executable if needed: chmod +x run/linux/start.sh
  3. Run the script: bash run/linux/start.sh
  4. On the first run, the script creates a Python virtual environment (.venv/) and installs all dependencies. This may take 2–5 minutes depending on internet speed and whether PyTorch needs to download.
  5. Subsequent runs detect if requirements.txt is unchanged (via SHA-256 hash stored in .req_hash) and skip reinstallation — startup takes only 2–4 seconds.
  6. Open http://localhost:8000 in your browser.
Dependency change detection: If you ever manually add packages to requirements.txt, delete run/linux/.req_hash to force a fresh pip install on next launch.

2.3 Starting on Windows

  1. Double-click run\windows\start.bat or run it from a Command Prompt.
  2. The script creates .venv\ and installs dependencies on the first run (same hash-based skip mechanism as Linux).
  3. A browser window opens automatically to http://localhost:8000.

2.4 First-Run Checklist

  • The database data/aura.db is created automatically if it does not exist.
  • ML model files are expected in the models/ directory. If missing, AURA still starts but the ML pipeline runs in a fallback mode (anomaly detection disabled).
  • The default admin password for Settings is admin. Change it immediately via Settings → Security.
  • For AI Analyst: ensure Ollama is running (ollama serve) and the desired model is pulled (ollama pull mistral). Alternatively, configure a Groq API key in Settings → Integrations.
  • The encryption key for the Groq API key is stored in .fernet.key in the project root. This file is machine-local and should never be committed to version control.
Security note: AURA is designed for local or trusted-network use. The web server binds to 0.0.0.0:8000 by default. Do not expose port 8000 to the public internet without adding reverse-proxy authentication.

7 · Trends Analysis

The Trends page applies multi-method statistical analysis to detect gradual degradation trends that may not yet register as anomalies in the Isolation Forest. This gives earlier warning of slow-developing faults (e.g., filter saturation, slow coolant leaks).

Controls

  • Location — select which ISS module to analyze
  • Readings — number of most recent readings to analyze (60 / 100 / 200)
  • Analyze button — runs the analysis immediately
  • Auto-refresh (30s) checkbox — automatically re-runs analysis every 30 seconds

7.1 Statistical Methods

Every parameter is evaluated against four independent tests simultaneously:

Mann-Kendall Trend Test
Non-parametric test for monotonic trend. Computes τ (tau) and p-value. A statistically significant trend (p < threshold, |τ| > advisory threshold) triggers at minimum an Advisory. Higher tau magnitude combined with direction toward a safety boundary escalates to Warning.
Sen's Slope
Estimates the rate of change per reading in physical units. Acts as a magnitude gate — a statistically significant trend must also show meaningful rate of change (configurable slope gate) to avoid advisories on imperceptibly slow drift.
CUSUM (Cumulative Sum)
Detects persistent upward or downward shifts by accumulating deviations from a baseline established from the first 15–40 readings. A CUSUM value exceeding the threshold indicates a sustained mean shift. Contributes one "signal" toward multi-signal confirmation.
Rolling Z-Score
Computes the z-score of the most recent N readings relative to a rolling mean/std window. High z-scores indicate recent values are statistically unusual relative to recent history. Two thresholds: standard (requires 2+ signals) and single-signal (very high z, fires alone).

7.2 Severity Levels

SeverityConditionInterpretation
NOMINALNo test signals fireParameter is stable
ADVISORYMK significant (p < 0.01, |τ| > 0.35) + slope gate + ≥2 total signalsSlow trend detected; monitor
WARNINGAdvisory conditions met AND trend direction is toward a safety boundaryParameter drifting toward out-of-spec; investigate
CRITICALMK highly significant (|τ| > 0.65) + all signals + rapid slopeImminent boundary breach; act now

Each trend card also displays the trend direction (increasing / decreasing / no trend), the current value with the nominal range, and the Sen's slope rate of change.

Multi-signal requirement: AURA requires at least two independent test signals to fire before issuing an Advisory. A single elevated z-score reading alone will not trigger an advisory unless it exceeds the single-signal threshold (default 4.5σ). This prevents false alerts from isolated noise spikes.

8 · Alerts

The Alerts page shows the historical log of all anomaly alert events. An alert is fired when the Isolation Forest flags a location as anomalous for a configurable number of consecutive ticks (default 10), subject to a cooldown period between repeated alerts for the same location.

Alert Firing Logic

  1. Isolation Forest flags a reading as anomalous (label = -1).
  2. The consecutive anomaly counter for that location increments.
  3. When the counter reaches Min Consecutive (default 10), check: has the cooldown period elapsed since the last alert for this location?
  4. If yes, fire the alert. Record severity based on Random Forest confidence: ≥ 85% RF confidence → CRITICAL; otherwise WARNING.
  5. The counter resets on any nominal reading.

Alert Severity

WARNINGSustained anomaly detected; RF confidence below CRITICAL gateAmber
CRITICALSustained anomaly + RF fault classification ≥ 85% confidenceRed

Alert Card Information

Each alert card displays:

  • Timestamp — when the alert was fired
  • Location — which ISS module triggered the alert
  • Severity — WARNING or CRITICAL
  • Fault type — the RF-classified fault name (if confidence was sufficient)
  • RF confidence — probability score of the fault classification
  • Sensor snapshot — the sensor readings at the time of the alert
  • Acknowledge button — marks the alert as reviewed

Filtering

  • Unacknowledged only checkbox — hide alerts that have already been acknowledged
  • Location filter — limit the list to a specific module
  • Acknowledge All — bulk-acknowledge all visible alerts
Alert badge: The red badge on the Alerts tab in the navigation bar shows the current count of unacknowledged alerts. It refreshes from the WebSocket on every new alert and polls the REST API every 30 seconds as a fallback.

9.1 Ollama (Local AI)

When Ollama is running locally (ollama serve), AURA connects to it at http://localhost:11434. Responses are streamed token-by-token and displayed in real time. Recommended models: mistral (fast, general purpose), llama3.1:8b (higher quality), llama3 (balanced).

Ollama is fully private — no data leaves your machine. Response latency depends on hardware (GPU recommended for models ≥ 7B parameters).

9.2 Groq (Cloud Fallback)

If Ollama is unavailable, AURA automatically falls back to the Groq API (if a key is configured in Settings → Integrations). Groq serves open models (Mixtral, LLaMA) on dedicated hardware with very low latency. Responses are also streamed.

Configure the Groq API key in Settings → Integrations → Groq API. The key is stored encrypted on disk using Fernet symmetric encryption.

9.3 Chat Memory

Chat history is persisted to browser localStorage so conversations survive page refreshes and tab closes. The AI receives the last N messages (configurable in Settings → Display → Max chat stored) as conversation history, enabling follow-up questions without re-stating context.

The Clear Chat button wipes both the visible chat and the stored localStorage history.

Example Questions

  • "Why is Node 2 showing anomalous readings?"
  • "What is the current CO₂ trend across all modules?"
  • "The LSTM is showing WARNING for US Lab — what should I check?"
  • "Summarize all active alerts and recommend priority actions."
  • "What is the RUL estimate for Joint Airlock and what does the DQN recommend?"
Context injection: Every message sent to the AI includes a full system state block. This means the AI always has current data — you do not need to paste readings or describe conditions manually. Simply ask in natural language.

10 · Maintenance Schedule

The Maintenance page provides two evidence-based schedules: a subsystem replacement schedule based on observed ISS ECLSS MTBF data, and a sensor calibration schedule tracking accumulated drift against calibration thresholds.

Mission Clock

The mission elapsed time counter in the page header shows how long AURA has been running since the configured mission start date (set in Settings → Display → Mission Start). This drives the MTBF life-fraction calculations.

Subsystem Replacement Schedule

Each subsystem card shows:

  • Subsystem name and related fault type
  • MTBF — mean time between failures in hours, sourced from NASA post-flight anomaly reports
  • % Life Used — mission elapsed hours ÷ MTBF
  • Hours Remaining to next expected service interval
  • Maintenance type — calendar-based (replace on schedule) or condition-based (replace when threshold breached)
  • Status badge — NOMINAL / DUE SOON / OVERDUE based on % life used

MTBF values are calibrated to actual ISS operational experience (e.g., OGS: 3,104 h actual vs. 8,437 h pre-flight estimate).

Sensor Calibration Schedule

The calibration table tracks drift accumulation for sensors with known drift rates (sourced from ICES papers and JSC-62802). Columns:

ColumnDescription
Sensor / ParameterName of the sensor and its parameter
Drift RateFraction of nominal span per week (from calibration literature)
Accumulated DriftTotal drift since last calibration baseline, as % of span
Weeks Until DueWeeks remaining before accumulated drift exceeds the 2% threshold
StatusOK / DUE SOON / CALIBRATE NOW
Note on TCCS: The Trace Contaminant Control System charcoal bed life is ~19,746 hours (2.25 years). There were no TCCS flight failures in 20+ years of ISS operations. A 90-day sampling cycle does NOT mean 90-day replacement — this is a common misconception. AURA uses the corrected MTBF value.

11 · Settings

All system configuration is accessible via the Settings tab. Settings are protected by a bcrypt-hashed admin password. All changes hot-reload into the running server — no restart is required. Settings are persisted to settings.json in the project root.

11.1 Authentication

When navigating to Settings, a login wall is displayed. Enter the admin password and click Unlock. A JWT token (30-minute TTL) is issued and stored in sessionStorage — it is automatically cleared when the browser tab is closed.

  • Default password: admin — change immediately via Security tab
  • Lock button — manually invalidates the session token and returns to login wall
  • The token is sent as a Bearer header on every Settings API request
  • If the server is restarted, all tokens are invalidated (new JWT secret generated at startup)

11.5 Fault Management

The Faults tab replaces the old twin sidebar controls with a more comprehensive interface:

  • Active Faults table — shows the current injected fault (if any) for each of the 7 locations, with a per-row Resolve button to clear that location's fault
  • Inject Fault form — select a location and fault type; a description panel shows the fault's precursor detection window and which sensor parameters it affects; click Inject Fault to apply
  • Clear All Injected Faults — removes all injected faults at once
Important: Injecting a fault only starts the drift — the ML pipeline may take several ticks to detect it depending on the fault type and current thresholds. Faults with fast drift rates (e.g., NH₃ Coolant Leak, O₂ Leak) produce anomaly signals within 1–3 minutes. Slow faults (e.g., Water Processor Failure, Trace Contaminant Filter Saturation) may take much longer.

11.10 Security

Change Admin Password

Enter the current password, then the new password twice (minimum 6 characters). Passwords are hashed with bcrypt before storage. The plaintext password is never stored or logged.

Revoke All Sessions

Clicking Revoke All Sessions generates a new JWT secret on the server. This immediately invalidates every active session token including the current one. You will be automatically logged out and returned to the login wall. Use this if you suspect an unauthorized session is active.

12 · Fault Type Reference

AURA simulates eight fault types derived from actual ISS ECLSS anomaly data. Each fault injects drift into specific sensor parameters at calibrated rates.

Cabin Leak
Precursor window: 8 hours
Affected parameters: Cabin pressure ↓, N₂ ↓, O₂ partial pressure ↓, CO₂ ↓, Humidity ↓, Temperature ↓, Airflow rate ↑
MTBF: 43,800 h (5-year mean)
Remediation: Systematic hatch isolation, ultrasonic leak detection, RTV sealant or fitting replacement
SMAC trigger: Cabin pressure < 14.0 psia or drop rate > 0.05 psia/hr sustained 2h
O₂ Generator Failure
Precursor window: 24 hours
Affected parameters: O₂ output rate ↓, O₂ partial pressure ↓, O₂ purity ↓, H₂ (ppm) ↑, Cabin pressure ↓, CO₂ ↑
MTBF: 3,104 h actual (pre-flight estimate was 8,437 h — 63% worse than expected)
Remediation: Inspect/replace cell stack; purge gas bubbles from electrolysis cells; check H₂ separator membrane
SMAC trigger: O₂ partial pressure < 19.5% or H₂ > 2% in O₂ stream
O₂ Leak
Precursor window: 4 hours
Affected parameters: O₂ partial pressure ↓, Cabin pressure ↓, O₂ (mass spec) ↓, Airflow rate ↑
MTBF: 8,760 h (~annual inspection cycle for plumbing seals)
Remediation: Inspect O₂ supply fittings and isolation valve seals; close O₂ isolation valve if leak confirmed
SMAC trigger: O₂ partial pressure > 23.5% or rising trend > 0.5%/hr
CO₂ Scrubber Failure
Precursor window: 4 hours
Affected parameters: CO₂ partial pressure ↑, CO₂ (mass spec) ↑, O₂ partial pressure ↓, Temperature ↑
MTBF: 4,380 h (~6-month major service per SSP-57003)
Remediation: Replace Zeolite 13X desiccant/sorbent beds; inspect selector valves; check heater elements
SMAC trigger: CO₂ partial pressure > 0.70% (5.3 mmHg); >1.0% is crew health limit
CHX Failure (Condensing Heat Exchanger)
Precursor window: 12 hours
Affected parameters: Humidity ↑, Temperature ↑, Bacterial/fungal count ↑, Airflow rate ↓
MTBF: 17,520 h (2-year mean; hydrophilic coating degrades over 2–4 years)
Remediation: Inspect and re-apply hydrophilic coating; clean microbial growth; check condensate separator disk
SMAC trigger: Cabin RH > 70% sustained > 48h or temperature > 27°C
Water Processor Failure
Precursor window: 48 hours
Affected parameters: Water purity ↓, Production rate ↓, Bacterial/fungal count ↑, H₂O ↓
MTBF: 3,850 h actual; sieve beds replaced annually on schedule
Remediation: Replace iodinated resin sieve beds; replace multifiltration beds when TOC > 3 mg/L; inspect ion exchange beds
SMAC trigger: TOCA TOC > 3 mg/L — immediate water processing halt required
Trace Contaminant Filter Saturation
Precursor window: 72 hours
Affected parameters: NH₃ ↑↑, CH₄ ↑, CO ↑↑, H₂ (ppm) ↑
MTBF: 19,746 h (~2.25 years; no TCCS flight failures in 20+ years of ISS ops)
Remediation: Replace activated charcoal bed and LiOH bed on saturation confirmation; 90-day sampling to track loading
SMAC trigger: NH₃ > 2.6 ppm (180-day SMAC) or any VOC above SMAC-1 limits
NH₃ Coolant Leak
Precursor window: 2 hours (fast-developing)
Affected parameters: NH₃ ↑↑↑, Temperature ↑, O₂ ↓, Cabin pressure ↑
MTBF: 17,520 h (~2-year mean for major EATCS loop leak events)
Remediation: Inspect pump module fittings and quick-disconnects; replace external pump module (EPM) if leak confirmed
SMAC trigger: NH₃ > 10 ppm requires PPE; > 25 ppm is emergency evacuation threshold

14 · ML / AI Systems Reference

14.1 Isolation Forest (Anomaly Detection)

The Isolation Forest (IF) is trained on nominal sensor data only. It operates on the complete 20-parameter sensor vector for a given location.

  • Output: Binary label — +1 (nominal) or -1 (anomalous)
  • Speed: Sub-millisecond inference; runs every tick
  • Principle: Anomalous points are easier to isolate by random feature splits (fewer splits needed)
  • Training: Fit on clean nominal data; model file: models/isolationForestModel.joblib
  • Role in pipeline: First-stage filter. Does not identify fault type — only flags statistically unusual readings

14.2 Random Forest Classifier (Fault Classification)

The Random Forest (RF) runs on every tick regardless of the IF label, providing fault classification across all 8 fault types plus a "nominal" class.

  • Output: Predicted fault class + confidence probability (0–1)
  • Bypass threshold: When RF confidence ≥ 0.85, the DQN uses the RF prediction directly rather than the raw state vector
  • Latch logic: 3+ consecutive ticks at ≥ 95% confidence for the same fault class triggers the fault latch popup
  • Training: Fit on labeled fault-drift data; model file: models/randomForestModel.joblib

14.3 LSTM Predictor (Failure Probability + RUL)

The LSTM (Long Short-Term Memory) network with multi-head attention processes a rolling window of recent sensor readings to produce two outputs:

Failure Probability

Scalar 0–1. The probability of imminent subsystem failure based on recent trends. Maps to NOMINAL / ELEVATED / WARNING / CRITICAL status words displayed on Dashboard cards.

Remaining Useful Life (RUL)

Estimated hours until failure, derived from FAULT_PRECURSOR_HOURS calibration data. Displayed in hours on Dashboard cards. Only meaningful when failure_prob > 0.2.

  • Rolling buffer: The LSTM maintains a per-location rolling buffer of the last N sensor readings. Predictions are not issued until the buffer is sufficiently filled.
  • Architecture: LSTM layers + multi-head attention + fully connected regression head
  • Model file: models/lstm_predictor.pt
Cold start: After clearing LSTM buffers or starting fresh, the LSTM will show no predictions until the rolling buffer accumulates enough readings (typically 30–60 ticks). This is expected behavior.

14.4 DQN Recommender (Remediation Actions)

The Deep Q-Network (DQN) maps the current system state to one of 11 possible remediation actions. It is trained via reinforcement learning with a reward structure that prioritizes restoring nominal conditions.

  • State vector: Current sensor readings, IF label, RF classification, LSTM failure_prob, and RUL for a location
  • Output: Action index + confidence. The action name is displayed on each Dashboard card and in the Alerts view.
  • RF bypass: When RF confidence ≥ DQN bypass threshold, the DQN receives the RF-identified fault as part of its state, improving action specificity
  • Fallback: If the DQN model is not loaded, a rule-based fallback maps RF fault class directly to the corresponding remediation action
  • Model file: models/dqn_recommender.pt

Settings Persistence

Settings are stored in settings.json at the project root. On server startup, settings_manager.py loads this file (or creates it from defaults) and applies all values to the running module variables via apply_to_main(), apply_to_dqn(), and apply_to_trend_detector(). No restart is required for threshold changes to take effect. The password bcrypt hash and encrypted Groq key are stored in this file; the Fernet encryption key is in a separate .fernet.key file.
